PhotoCast Pipeline Reference

Internal developer reference — Last updated: 2026-04-10

1

Overview

End-to-end pipeline: Open-Meteo weather + aerosol data → weather triage (zero-cost pre-flight) → Claude evaluation (colour scoring per location) → H2/PostgreSQL persistence → role-gated API. Parallel aurora pipeline polls NOAA SWPC every 5 min, triages by cloud, and scores viable dark-sky locations via a single Claude call. Eight scheduled jobs orchestrated by a DB-backed dynamic scheduler.

▶

PhotoCast is a photographer's forecast system. Its core value proposition: for each configured location, it fetches weather, aerosol, tidal, and solar geometry data, assembles a rich atmospheric snapshot, and asks Claude to score the sunrise/sunset colour potential on two axes (Fiery Sky 0–100, Golden Hour 0–100) plus a 1–5 star rating.

The system operates on four tiers of Claude usage:

  • Daily Briefing — zero-Claude-cost weather triage producing GO/MARGINAL/STANDDOWN verdicts per location, plus a single Haiku call for "best bet" recommendations and parallel Haiku calls for per-region glosses.
  • Briefing Gloss — per GO/MARGINAL region, a single Haiku call (max 64 tokens) generates a ≤10-word plain-English sentence explaining the key reason for the verdict. Parallel via virtual threads with semaphore (max 10 concurrent).
  • Per-Location Forecast — full Claude evaluation (Haiku/Sonnet/Opus configurable per run type) with directional cloud sampling, cloud approach risk, storm surge, and inversion scoring.
  • Aurora Photography — NOAA SWPC polling with state-machine alert management, weather triage, and a single Claude call to score all viable dark-sky locations.

All Claude calls route through a single AnthropicApiClient bean with Resilience4j @Retry + @CircuitBreaker. Token usage and costs are tracked per API call in the api_call_log table.

2

Weather Triage

Two independent triage systems: BriefingVerdictEvaluator for the daily briefing (granular multi-metric checks) and WeatherTriageEvaluator for forecast runs (fast 3-check gate before Claude). Both produce verdicts that can skip expensive Claude calls. Cost: zero (Open-Meteo only).

2a. Briefing Verdict Evaluator

▶

Used by BriefingSlotBuilder to assign a GO / MARGINAL / STANDDOWN verdict to each location × solar-event slot in the daily briefing.

Open-Meteo Fields Fetched

Main forecast batch (per location):

cloud_cover_low, cloud_cover_mid, cloud_cover_high, visibility,
wind_speed_10m, wind_direction_10m, precipitation, weather_code,
relative_humidity_2m, surface_pressure, shortwave_radiation,
boundary_layer_height, temperature_2m, apparent_temperature,
precipitation_probability, dew_point_2m, pressure_msl, wind_gusts_10m

Horizon cloud-only batch (113 km east for sunrise, west for sunset):

cloud_cover_low, cloud_cover_mid, cloud_cover_high

Verdict Determination Cascade

Each check can only demote the verdict (never promote). Applied in this exact order:

#CheckThresholdEffect
1Base verdictSee table below
2Mid-cloud demotion≥80% → STANDDOWN; ≥60% → MARGINALCan only demote
3Building cloud trendPeak ≥40% AND event > earliestGO → MARGINAL only
4Clear-all-layersLow <15% AND Mid <15% AND High <15%GO → MARGINAL only
5Horizon low cloud≥70% → STANDDOWN; ≥40% → MARGINALCan only demote
6Coastal tideTide not alignedOverride → STANDDOWN

Base Verdict Thresholds

MetricSTANDDOWNMARGINALGO
Low cloud>80%>50%≤50%
Precipitation>2.0 mm>0.5 mm≤0.5 mm
Visibility<5,000 m<10,000 m≥10,000 m
Humidity—>90%≤90%

Standdown Reason Priority Order

When verdict is STANDDOWN, the first match becomes the reason:

  1. HEAVY_CLOUD — lowCloud >80%
  2. OVERCAST — midCloud ≥80%
  3. RAIN — precip >2.0 mm
  4. POOR_VISIBILITY — visibility <5,000 m
  5. BUILDING_CLOUD — building trend detected
  6. TIDE_MISMATCH — coastal tide not aligned
  7. CLEAR_SKY — all layers <15% (no canvas)
  8. SUN_BLOCKED_HORIZON — horizon demotion
  9. POOR_CONDITIONS — fallback

Region Rollup

For a region, the verdict is rolled up by excluding STANDDOWN slots: if all viable slots are GO → GO; some viable but none GO → MARGINAL; all STANDDOWN → STANDDOWN.

Data Flow

BriefingService.refreshBriefing() ↓ Load enabled colour locations ↓ OpenMeteoClient.fetchForecastBriefingBatch() → main weather OpenMeteoClient.fetchCloudOnlyBatch() → horizon cloud (113 km) ↓ Per location × date × event: BriefingSlotBuilder.buildSlot() → determineVerdict() → apply demotions → buildFlags() → GO / MARGINAL / STANDDOWN ↓ Group into BriefingDay → BriefingEventSummary → BriefingRegion ↓ BriefingBestBetAdvisor.advise() → Claude picks (single Haiku call) ↓ Cache in AtomicReference + daily_briefing_cache DB table ↓ GET /api/briefing → DailyBriefingResponse

2b. Forecast Run Weather Triage

▶

WeatherTriageEvaluator is a fast heuristic gate in the per-location forecast pipeline. It runs before any Claude call and checks three conditions. If any match, the location gets a canned rating=1 entity and Claude is skipped.

CheckThresholdReason
Solar low cloud>80%Sun blocked at horizon
Precipitation>2.0 mmActive rain

Additionally, if TIDE_ALIGNMENT optimisation is enabled for SEASCAPE locations, a tide-misalignment check can also trigger a canned skip.

2c. Aurora Weather Triage

▶

WeatherTriageService is aurora-specific. Uses a 3-point northward transect (50/100/150 km north) to check cloud cover at each location.

ParameterValue

Rejected locations get auto-assigned 1-star with no Claude call.

3.1 Briefing Best-Bet Advisor

▶

Identifies 2 best photographic opportunities from the briefing rollup. Single call after briefing triage completes.

Service & Method

BriefingBestBetAdvisor.advise(List<BriefingDay>, Long jobRunId, Map<String,Integer> driveMap)

System Prompt Summary

Role: "photography forecast advisor for PhotoCast". Identify 2 picks: (1) best overall across all regions/events, (2) best alternative on a different day/event/region. Evaluation factors: GO regions with most clear locations rank highest, tide alignment as strong differentiator, lunar types (King > Spring > Regular), statistical sizes, storm surge as force multiplier, aurora integration, comfort, drive times, day-of-week preference. Sooner events preferred if quality equal.

User Message Structure

  • Current UTC time
  • Event-based JSON rollup (next 6 solar events max):
    • Event ID, date, day name, event time
    • Per-region verdict, go/marginal/standdown counts
    • Temperature, apparent temperature, wind speed, weather code
    • Tide counts: lunarKingTideCount, lunarSpringTideCount, extraExtraHighCount, extraHighCount, tideAlignedCount
    • Coastal/inland location counts
    • Storm surge boost flag
  • Optional: Aurora event (alert level, Kp, dark sky / clear location counts)
  • Forecast window, valid events, valid regions

Output Schema

{
  "picks": [
    {
      "rank": 1,
      "headline": "max 15 words, punchy",
      "detail": "2 sentences max, 40 words max",
      "event": "<eventId from validEvents>",
      "region": "<region name from validRegions>",
      "confidence": "high|medium|low"
    }
  ]
}

Fallback

On parse failure: returns mechanical headline derived from best GO region. Validates picks against validEvents, validRegions, validDayNames; invalid picks are dropped and survivors re-ranked.

Access Control

Triggered internally by BriefingService — results visible to all authenticated users via GET /api/briefing. Model comparison endpoint is ADMIN-only.

3.2 Briefing Region Gloss

▶

Per GO/MARGINAL region, generates a ≤10-word plain-English sentence explaining the key reason for the verdict. Parallel calls via virtual threads.

Service & Method

BriefingGlossService.generateGlosses(List<BriefingDay> days, Long jobRunId)

Called from BriefingService.refreshBriefing() after hierarchy assembly, before caching. STANDDOWN regions are skipped (no API call).

Model & Tokens

ParameterValue

You are a photography forecast assistant. Given weather and tide data for a
region at a solar event, write a single plain-English sentence of 10 words or
fewer explaining the key reason for the verdict. No quotes, no punctuation
other than what the sentence needs. Be specific about cloud conditions.
Examples: "High cirrus canvas — good colour potential",
"Clear all layers — flat light, nothing to catch colour",
"Mid-level cloud building — risk of blanket by event time".

  "event": "sunset",
  "date": "2026-04-10",
  "verdict": "GO",
  "cloudLow": 15,          // median % across GO slots (or non-STANDDOWN)
  "cloudMid": 25,
  "cloudHigh": 40,
  "clearAllLayers": false,  // flag presence check
  "buildingTrend": false,
  "tideAlignedCount": 0,
  "hasKingTide": false,
  "hasSpringTide": false,

  "goCount": 1,

  "totalLocations": 1
}

Output Format

Plain UTF-8 text (no JSON schema). Response stripped of whitespace and stored directly as the gloss field on BriefingRegion. No length validation beyond the 64-token cap.

Parallelism

MechanismDetail
ExecutorVirtual threads via CompletableFuture.runAsync()
Concurrency capSemaphore(10) — max 10 concurrent Claude calls
Wait strategyCompletableFuture.allOf().join() — blocks until all complete

Error Handling

  • Per-call: Exception caught, logged with duration and error message via jobRunService.logApiCall(). Gloss field stays null for that region. No re-throw.
  • Global: Entire generateGlosses() wrapped in try-catch. On failure, returns the original days list with all gloss fields null.
  • Retry/CB: Delegated to AnthropicApiClient Resilience4j decorators (max 4 attempts, exponential backoff).

Storage & API Exposure

Not persisted to DB. Gloss is computed at cache-time, stored in-memory on BriefingRegion.gloss (nullable String) within the DailyBriefingResponse, and serialized to JSON via GET /api/briefing. Regenerated on each briefing refresh.

Immutable record reassembly: BriefingGlossService reconstructs new BriefingDay / BriefingEventSummary / BriefingRegion instances with populated gloss values (Java records are immutable).

Access Control

Visible to all authenticated users via the briefing endpoint. Model configurable via Admin UI (RunType.BRIEFING_GLOSS).

Forecast Evaluation Pipeline

Command pattern: ForecastCommand → ForecastCommandFactory → ForecastCommandExecutor. Three-phase pipeline: batch pre-fetch (2 API calls instead of 2N) → triage (parallel, filters ~80% of unsuitable conditions) → sentinel sampling → stability filter → full Claude evaluation (parallel, bulkhead=8).

Inversion Scoring & Storm Surge

▶

Inversion Score (0–10)

Gate: elevation ≥300m AND overlooks_water = true. Calculated by InversionScoreCalculator:

FactorPointsLogic
Temp–dew gap0–4≤1°C=4, ≤2°C=3, ≤4°C=2, ≤6°C=1
Wind speed0–3≤1.5 m/s=3, ≤3=2, ≤5=1
Humidity0–2≥90%=2, ≥80%=1.5, ≥70%=1
Low cloud0–120–60%=1, >60%=0.5

Storm Surge Model

Calculated at high tide time (not solar event time) by StormSurgeService:

  • Inverse barometer: 0.01 m × (1013.25 − actual_hPa)
  • Wind setup: Bulk aerodynamic formula with 0.5 calibration factor and shore-normal projection
  • Risk levels: <0.05m = INSIGNIFICANT (suppressed), 0.05–0.15m = LOW, 0.15–0.30m = MODERATE, 0.30–0.50m = HIGH, ≥0.50m = EXTREME

Upper-bound estimate. No wave setup modelled. Calibration logging enabled for empirical tuning.

Aurora Pipeline

NOAA SWPC polling every 5 min + Met Office scraping hourly. Dual-path: forecast lookahead (any time) + real-time (night only). State machine (IDLE↔ACTIVE) with escalation/de-escalation. Single Claude Haiku call for all viable locations. Conservative: Kp latitude hard caps, dual-signal alert logic, lenient cloud triage.

Alert Level Derivation

▶

AuroraOrchestrator.deriveAlertLevel() uses dual-signal logic:

effectiveKp = max(currentKp, maxForecastKp_6h_ahead)

if effectiveKp >= 7.0 -> STRONG
if effectiveKp >= 5.0 OR ovation% >= 20% -> MODERATE
if effectiveKp >= 4.0 -> MINOR
else -> QUIET

DISABLED_BY_CONFIG

Three aurora jobs (aurora_polling, met_office_scrape, aurora_batch_evaluation) auto-disable when aurora.enabled=false. UI shows grey status pill with config source. All pause/resume/trigger operations reject with IllegalStateException.

Validation

  • Cron expressions validated via CronExpression.parse()
  • Fixed delay minimum: 10,000 ms (10 seconds)
  • Concurrency: ReentrantLock protects all scheduling operations

ExchangeRateService

Not an in-memory cache — uses exchange_rate DB table with rateDate primary key. On miss, fetches from Frankfurter API (ECB data) and persists. Fallback: most recent DB rate on API failure. Exchange rates can be up to 24h old; low impact since rates change gradually.
